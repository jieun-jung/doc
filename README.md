# 코딩 컨벤션 

## 1. 범위

| 구분 | 설명 |
|------|------|
| 저장소 | `pnpm` 워크스페이스 모노레포 (`packages/`, `services/`) |
| 주 스택 | TypeScript, React 18, Vite |
| UI·문서 | Storybook, 디자인 시스템 패키지(`packages/*`) |

---

## 2. 언어 & 문자열

- **UI 문구·주석·커밋 메시지(팀 정책 시)**: 한국어 가능. 코드 식별자(변수·파일명)는 **영어** 유지.
- **사용자에게 보이는 문자열**은 하드코딩보다 **상수·i18n 레이어**로 모을지 팀에서 결정.

---

## 3. TypeScript

- `any`는 **불가피한 경우에만** 사용하고, 그 경우 **근거를 주석 한 줄**로 남기거나 이슈로 추적.
- **공개 API**(export 함수·컴포넌트 props)는 타입을 **명시**하는 것을 권장.
- `enum` 대신 **union 타입 + `as const`** 를 우선 검토(트리셰이킹·일관성).
- API 응답·도메인 타입은 **`src/api-types` 등 전용 디렉터리**에 두고, 화면 내부에서 재정의하지 않기.

---

## 4. 네이밍

| 대상 | 규칙 | 예 |
|------|------|-----|
| React 컴포넌트 파일 | PascalCase | `SettleHistory.tsx` |
| 일반 모듈·유틸 파일 | camelCase 또는 kebab-case **한 가지로 통일** | `formatDate.ts` |
| 컴포넌트 | PascalCase | `GridSection` |
| 훅 | `use` 접두사 | `useSettleList` |
| boolean 변수 | `is` / `has` / `should` 등 의미 접두사 | `isLoading` |
| 이벤트 핸들러 | `handle` + 주어 | `handleRowClick` |
| API 타입 파일 | 백엔드/스펙과 맞춘 접두사·접미사 유지 | `fapi_settleHistory.ts` |

---

## 5. 디렉터리 & 파일 구조

- **기능 단위(feature)로 폴더**를 나누고, 해당 화면 전용 컴포넌트·훅은 **같은 트리에 colocate**.
- `pages/` 또는 `routes/`의 **한 페이지 = 한 폴더**(`index.tsx`, `section/`, `options.tsx` 등) 패턴을 유지.
- `index.ts`에서만 re-export 하는 **배럴 파일**은 팀 합의 하에만 사용(순환 참조 주의).

---

## 6. React

- **함수 컴포넌트**만 사용. 클래스 컴포넌트 신규 작성 금지.
- 재사용 로직은 **커스텀 훅**으로 분리(`use*`).
- **리스트/키**: 안정적인 `id`를 `key`로 사용. 인덱스 키는 정렬·필터가 없을 때만.
- **AG Grid 등**: 컬럼 정의·이벤트 핸들러는 **메모이제이션**(`useMemo`/`useCallback`) 필요 여부를 검토해 불필요한 리렌더 방지.

---

## 7. import 순서 (권장)

1. 외부 패키지 (`react`, `ag-grid-community`, …)
2. 내부 패키지·별칭 (`@/`, `@defign/`, …)
3. 상대 경로 (`./`, `../`)
4. 타입만 import 할 때는 `import type` 사용

한 그룹 안에서는 **알파벳 순** 정렬을 권장(Prettier 플러그인 사용 시 자동화 가능).

---

## 8. 스타일

- 프로젝트에 이미 있는 방식(**CSS Modules / SCSS / 디자인 토큰**)을 따름. 신규로 여러 방식을 섞지 않기.
- **인라인 `style`** 은 동적 값 등 예외적 경우에만.

---

## 9. 포맷 & 린트

- **저장 시 포맷**(Prettier)과 **ESLint**를 통과한 코드만 PR에 포함.
- `eslint-disable`는 **한 줄·근거**를 남기고, 가능하면 범위를 최소화.

---

## 10. 테스트 & Storybook

- 공통 UI·순수 유틸은 **단위 테스트 또는 스토리**로 회귀 방지.
- 스토리 네이밍·controls는 팀 템플릿에 맞출 것.

---

## 11. Git & PR

- 브랜치·커밋 규칙은 팀 정책에 따름(예: Conventional Commits).
- PR 설명에는 **무엇을·왜**를 한국어로 간결하게.

---

## 12. 보안·품질

- 시크릿·토큰을 소스에 넣지 않음. 환경 변수·시크릿 매니저 사용.
- `console.log`는 개발용으로만; 머지 전 제거 또는 로거로 통일.
